这是一个非常深刻的切入点。在实施延迟退休政策后，退休年龄不再是一个固定的常数（如 60 或 55），而是一个基于**出生年月**动态计算的变量。

在 Palantir 本体论设计中，这种“动态计算”绝不应该写在 DRL 规则里，而应该封装在 **`UnemploymentPolicy`**（作为计算引擎）或 **`UnemployedPerson`**（作为属性转换）中。

以下是针对延迟退休逻辑重构的 DRL 修改清单及 GitHub Copilot 指令。

---

### 一、 DRL 文件重构核心：从“比对数值”转向“调用函数”

由于退休年龄变成了动态值，DRL 必须从“硬编码比对”彻底转向**“语义化方法调用”**。

#### 修改清单：

1. **删除所有固定年龄逻辑**：删除 DRL 中类似 `age >= 60` 或 `age >= 55` 的条件。
2. **引入策略对象计算**：规则应询问 `UnemploymentPolicy` 对象：“根据这个人的出生年月，他的法定退休年龄是多少？”
3. **判定临近退休**：调用 `UnemployedPerson` 预计算好的 `isNearRetirement` 标志。

---

### 三、 最终版 DRL 逻辑结构预览

修改后的 DRL 将变成一个**纯逻辑网格**，不包含任何具体的年龄数字：

```drl
// 情形三：临近退休人员资格校验
rule "Unemployment_Scenario_3_DynamicRetirement"
    when
        $policy : UnemploymentPolicy()
        // 核心：isNearRetirement 内部已集成了延迟退休的动态计算逻辑
        $p : UnemployedPerson(
            isDomicileShenzhen == true, 
            isNearRetirement($policy) == true 
        )
        $timeline : UnemploymentTimeline(personId == $p.id)
        $latest : InsuranceRecord(
            this == $timeline.latestRecord,
            isInvoluntaryStop == true
        )
    then
        insert(new EligibilityResult($p.getId(), "SCENARIO_3", "符合延迟退休背景下的临近退休条件"));
end

```

---

### 四、 为什么这样设计符合 Palantir 本体论？

1. **解耦政策与逻辑**：延迟退休的算法（比如每 4 个月延 1 个月）属于**政策元数据**，应存在于 `UnemploymentPolicy` 类中。如果国家再次微调政策，你只需要改 Java 类里的一个算法函数，**不需要修改任何 `.drl` 文件**。
2. **数据孪生一致性**：`UnemployedPerson` 对象作为自然人的“数字孪生”，其 `isNearRetirement` 属性能够根据最新的 `UnemploymentPolicy` 实时反映其状态。
3. **GitHub Copilot 友好度**：当你给 Copilot 定义了清晰的 `UnemploymentPolicy` 接口后，它在编写规则时会自动联想并调用 `calculateStatutoryRetirementAge()`，从而避免生成错误的硬编码代码。

**建议：** 在你的 `UnemploymentPolicy` 中，可以增加一个 `List<RetirementMapping>`，存储不同出生时段对应的退休月数，这样 Copilot 就能帮你写出最准确的阶梯式计算逻辑。
